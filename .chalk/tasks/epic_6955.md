The CTO's vsock mandate makes Go a permanent part of the codebase: the guest-side vsock client
must be a static Go (or Rust) binary — Node has no native AF_VSOCK — built in our CI and covered
by the same cosign signing as the reviewer image. That raises the CTO's follow-on question this
epic answers and then implements: **what else should move to Go, and how do we manage the
migration while leveraging the existing TypeScript tests?**

Working recommendation (to be ratified in the scope-decision task):

- **Adopt Go only at process boundaries where Node is the wrong tool** — syscall-level work and
  static binaries that must run where Node doesn't exist:
  1. Guest-side vsock client (mandated) — ships inside the signed reviewer image.
  2. Install-time KVM/tier preflight probe (open /dev/kvm + KVM_CREATE_VM ioctl — raw ioctls in
     Node need a native addon anyway, and the installer runs before Node is provisioned).
  3. Host-side per-VM vsock↔gateway forwarder — only if the chosen VMM exposes the guest vsock
     as something Node can't serve; with hybrid-vsock (host side = unix socket) Node may
     suffice. Decide in the forwarder task; sharing one Go module with the guest client is the
     tiebreaker.
- **Do NOT migrate the orchestrator or gateway business logic this round** (webhook/queue/
  GitHub API/diff/findings/anchoring/publisher, gateway keystore). It is mature, tested
  TypeScript sitting on the Octokit ecosystem; a rewrite adds regression risk with zero
  isolation gain — these are trusted host-side processes either way. The gateway is the only
  plausible longer-term candidate (small proxy, static-binary deployment win); explicitly
  deferred, own decision later.
- **Migration mechanics = strangler pattern at process boundaries.** Go components are separate
  binaries with narrow contracts (argv/env/exit codes, sockets) — never in-process rewrites.
  Existing TS integration/e2e tests already exercise the pipeline *through* those process
  boundaries, so they become the cross-language contract harness and keep working unchanged
  when a Node piece is swapped for a Go binary. Go unit tests cover only new Go-internal logic;
  golden/contract tests (like the M8 floor-invariant flag test) pin behavior across swaps.
- **One Go module** (e.g. go/ workspace dir), CGO_ENABLED=0 reproducible static builds for
  amd64+arm64 in the existing release CI, cosign-signed; binaries baked into the reviewer image
  are additionally covered by the image digest pin + signature.
