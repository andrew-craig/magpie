**Language decision: Rust, not Go.** The original CTO mandate said the guest-side vsock client
must be a static "Go (or Rust)" binary (Node has no native AF_VSOCK); the M8-A1 spike resolved the
open Go-vs-Rust choice in favour of **Rust** on evidence, and this epic now adopts Rust as the
native-code language. Rationale (spike commits `f47eaf3`, `823a4dd`; see
`spike/m8-a1/frontend-investigation.md`):

- **The host-side micro-VM launcher is net-new and forces the choice.** Adopting the direct-libkrun
  front-end (M8-A1 finding) means we call libkrun's C ABI ourselves â€” `krun_add_vsock(ctx,0)` for
  provable no-network, `krun_add_vsock_port2` for the per-VM gateway channel, `krun_setuid`,
  `krun_set_vm_config`. That is FFI into a C library whose entry point (`krun_start_enter`) does
  not return â€” hostile to Go's runtime, and cgo would negate Go's static-binary/cross-compile
  advantage. Rust binds the C ABI cleanly and matches libkrun's own language (relevant if we ever
  patch it). C was the only other candidate; Rust wins on memory safety for a privileged component.
- **The guest-side client is proven in Rust.** A static musl `AF_VSOCK` client (389 KB, fully
  static, `libc` crate only) was built and did a full guestâ†”host vsock round-trip through
  `krun_add_vsock_port2` with TSI off (spike `f47eaf3`). It satisfies every requirement the Go
  mandate was written for, so there is no reason to run a second native language for it.
- **Net effect: a TS + Rust two-language stack**, not TS + Go + Rust. Fewer signed-artifact
  toolchains and one cross-compile/supply-chain lane is itself a security argument for an
  isolation product. (If the team lacks Rust fluency, keeping Go for the *guest client only* is a
  legitimate staffing fallback â€” but the launcher must still be Rust/C, never Go+cgo. Flag for CTO.)

Scope (ratify in RUST-1 / `decision_aa2d`):

- **Adopt Rust only at process boundaries where Node is the wrong tool** â€” syscall-level work,
  libkrun FFI, and static binaries that must run where Node doesn't exist:
  1. Guest-side vsock client (mandated) â€” ships inside the signed reviewer image. **Proven.**
  2. Host-side micro-VM launcher (net-new) â€” links libkrun, the FFI component above.
  3. Install-time KVM/tier preflight probe (open /dev/kvm + KVM_CREATE_VM ioctl â€” raw ioctls in
     Node need a native addon anyway, and the installer runs before Node is provisioned).
  4. Host-side per-VM vsockâ†”gateway forwarder â€” with hybrid-vsock the host side is a plain unix
     socket Node *could* serve, but if it lives in the same process as the launcher (which links
     libkrun) it is naturally Rust too. Decide in the forwarder task (`task_b3f7`).
- **Do NOT migrate the orchestrator or gateway business logic this round** (webhook/queue/
  GitHub API/diff/findings/anchoring/publisher, gateway keystore). It is mature, tested
  TypeScript sitting on the Octokit ecosystem; a rewrite adds regression risk with zero
  isolation gain â€” these are trusted host-side processes either way. The gateway is the only
  plausible longer-term candidate (small proxy, static-binary deployment win); explicitly
  deferred, own decision later.
- **Migration mechanics = strangler pattern at process boundaries.** Rust components are separate
  binaries with narrow contracts (argv/env/exit codes, sockets) â€” never in-process rewrites of the
  TS logic. (The launcher does link libkrun's C ABI, but that is a standalone native binary linking
  a C library, not FFI *into* the Node process â€” the boundary rule holds.) Existing TS
  integration/e2e tests already exercise the pipeline *through* those process boundaries, so they
  become the cross-language contract harness and keep working unchanged when a Node piece is
  swapped for a Rust binary. Rust unit tests cover only new Rust-internal logic; golden/contract
  tests (like the M8 floor-invariant flag test) pin behavior across swaps.
- **One cargo workspace** (e.g. `rust/` dir alongside the npm workspaces), reproducible static
  musl builds for amd64+arm64 in the existing release CI, cosign-signed; binaries baked into the
  reviewer image are additionally covered by the image digest pin + signature.
