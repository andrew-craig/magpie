Wave 2 (needs M4-A's gateway). Orchestrator-side key lifecycle per PLAN.md §5.

- New module (e.g. gateway.ts) in packages/orchestrator: mint a fresh LiteLLM virtual key before each review run — scoped to the configured model, with a per-job spend/request budget from config — and revoke it during job cleanup (success, failure, and timeout paths alike; cleanup must be best-effort and never mask the job result).
- This budget is the HARD cost cap (Pi has no --max-turns/budget flag): the gateway cuts the job off at its limit no matter what the agent does. Surface a budget-exhausted run as a review failure, not silence.
- Config: gateway base URL, management/master key (host-only secret, MAGPIE_* env — must be stripped from the reviewer env like the other secrets), per-job budget. Document in config.example.toml.
- Unit-testable with a fake gateway HTTP endpoint.

Done when: every job mints its own key, the key is revoked on all exit paths, and a job that exceeds its budget terminates with a clear failure comment.

## Review (M4-B implementation)

Branch: `m4-gateway`. Implemented the orchestrator-side per-job virtual-key lifecycle against the already-merged M4-A gateway (`packages/gateway`). Did not touch `packages/gateway/**` (M4-A), container env / base-URL rewiring (M4-C), or networking/scripts (M4-D).

### As-built `gateway.ts` interface
- `mintGatewayKey(gateway, { model?, budgetUsd, ttlSeconds }): Promise<{ id, key }>` — `POST {baseUrl}/admin/keys` with `Authorization: Bearer <masterKey>`; parses `{ id, key }` via a strict zod schema. THROWS on unreachable gateway, non-201, or unexpected/unparseable body (a job that can't get a key can't run — mirrors `github.ts` mint).
- `revokeGatewayKey(gateway, id, logger?): Promise<void>` — `DELETE {baseUrl}/admin/keys/:id` (id url-encoded). BEST-EFFORT, NEVER THROWS: any failure is logged and swallowed (mirrors `orphan-cleanup.ts` / `reviewer.ts` `output.cleanup()`), so a revoke failure can never mask a job outcome.
- `mintGatewayKeyFromConfig(config)` / `revokeGatewayKeyFromConfig(config, id, logger?)` — config adapters (mirror `mintInstallationTokenFromConfig`). Mint scopes the key to `config.llm.model`, budgets `config.gateway.perJobBudgetUsd`, TTL = `limits.jobTimeoutSeconds + gateway.ttlMarginSeconds`.
- SECURITY: master key and minted key are only ever placed in the request header — never in a URL, argv, log line, or thrown-Error message (unit-tested).

### Pipeline wiring (`pipeline.ts`)
- Step 2a: mint one gateway key per job, right after the installation token, before the workspace. Wrapped in an OUTER `try/finally` around the existing workspace `try/finally`, so the key is revoked on EVERY exit path: success, review-failed result, thrown error (incl. `makeWorkspace` throwing), and abort/timeout (runReview resolves `aborted` → runJob returns → finally revokes). Revoke is best-effort (never throws) so it never changes the outcome.
- Two new `PipelineDeps` seams (`mintGatewayKey`, `revokeGatewayKey`) default to the real config adapters; `index.ts` passes none, so production uses the real gateway.
- Note on the queue's separate `cleanupJob` (timeout backstop): it can't revoke by id (the random key id isn't derivable from the `JobDescriptor`), but it doesn't need to — `runJob`'s own `finally` always runs on timeout because `runReview` resolves promptly on abort. The only path that skips it is a hard process kill (`kill -9`), which is the accepted orphan scenario: gateway keys are TTL-capped and lost on gateway restart anyway (per M4-A's in-memory-store design).

### Config (`config.ts`, `config.example.toml`, `.env.example`)
- New `[gateway]` section: `base_url` (default `http://127.0.0.1:4100`), `per_job_budget_usd` (default `0.50`), `ttl_margin_seconds` (default `120`).
- New required env secret `MAGPIE_GATEWAY_MASTER_KEY` → `config.secrets.gatewayMasterKey` (must match the gateway process's own `MAGPIE_GATEWAY_MASTER_KEY`). Namespaced `MAGPIE_*`, so `reviewer.ts`'s existing `MAGPIE_`-prefix env strip already removes it from the docker-client/container env (verified).
- Did NOT remove `MAGPIE_LLM_API_KEY` handling — that removal + container base-URL/key rewiring is M4-C's job. Clean seam flagged: `reviewer.ts` line ~254 sets `env.OPENROUTER_API_KEY = config.secrets.llmApiKey`; M4-C swaps that for the per-job minted virtual key and points the container's base URL at the gateway proxy plane.

### Budget-exhaust → failure surfacing
Confirmed via the existing reviewer path: when the gateway refuses an over-budget request with 402, Pi's model call fails, Pi still exits 0 with a final assistant turn carrying `stopReason: "error"` / `errorMessage`, and never calls `report_findings`. `reviewer.ts`'s "no findings file" branch already detects this (`last.stopReason === "error" || last.errorMessage`) and returns `{ ok: false, reason: "pi review failed: <detail>" }`, which the pipeline publishes via `publishReview`'s failure-comment path — an intelligible "could not complete a review" comment, not silence. No new parsing needed.
