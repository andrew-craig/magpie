Remove the last secret from the reviewer container and make its egress default-deny. This is PLAN.md milestone 4 and PLAN.md §5 (Egress control: credential-injecting LLM gateway).

Stand up a host-side gateway (own unprivileged user) that holds the single real OpenRouter key and exposes an OpenAI-compatible endpoint. Per the CTO deviation (see PLAN.md §5 deviation box), this is a purpose-built OpenRouter-only TypeScript proxy (`packages/gateway`, `@magpie/gateway`) with an in-memory key store — NOT LiteLLM, and no Postgres/DB. Point Pi at it via a provider base-URL override. For every job the orchestrator mints a short-lived, budget-capped per-job virtual key before the run and revokes it on cleanup — this is also the hard cost cap Pi itself lacks. Put the container on a dedicated magpie-net bridge with host iptables default-deny so the gateway is the container's ONLY reachable destination (hostname-based filtering via the gateway, NOT an IP allowlist — see §5 on why CDN IP allowlisting is inadequate). GitHub must never be reachable from the container or the gateway.

As-built note (M4 complete): Pi 0.80.3 has no `OPENAI_BASE_URL` env override, so the base-URL redirect is delivered as a `~/.pi/agent/models.json` provider override written by the container entrypoint from the `OPENAI_BASE_URL` env var (`--provider openrouter` unchanged). See closed tasks M4-A..M4-E for the full record.

Definition of done: the container env contains no long-lived provider key (only the per-job virtual key); a review works end-to-end through the gateway; the entry script fails closed if any host other than the gateway is reachable or a long-lived key is present; a leaked virtual key is worthless (short-lived, budget-capped, revoked on cleanup).
