Serialize review jobs with bounded concurrency and a hard timeout so a stuck or runaway job can't wedge the service.

Context: In-process p-queue with CONCURRENCY 2 (from config) and a hard PER-JOB TIMEOUT (default 10 min). On timeout the job must be aborted and its workspace cleaned up. Losing queued jobs on a crash is acceptable for personal use (a re-push re-triggers review) — no Redis/BullMQ in this milestone.

Scope:
- A queue wrapper around p-queue with configurable concurrency.
- Enqueue a job descriptor (from event filtering) that runs the job pipeline (clone -> diff -> Pi -> post; wired in the integration task).
- Enforce a per-job wall-clock timeout that cancels the job and triggers cleanup (workspace removal; kill the Pi host process if still running).
- Structured logging per job: start, finish, duration, outcome (success/failed/timed-out/skipped).
- Basic dedup hook is a nice-to-have (replace a not-yet-started job for the same PR); full kill-and-requeue on newer SHA is a later milestone.

Acceptance criteria:
- No more than concurrency jobs run at once.
- A job exceeding the timeout is aborted and its workspace cleaned up.
- Each job logs a clear start/finish with outcome.

Dependencies: task_60fc (scaffolding), task_9c52 (concurrency + timeout config).

## Review (tech lead)
Implemented by sonnet subagent; reviewed + integration-verified by tech lead.
- Files: packages/orchestrator/src/queue.ts (+ queue.test.ts). Dep added: p-queue ^8.1.1.
- Approach: `JobQueue` wraps p-queue (concurrency cap). Per-job timeout armed at job START (not enqueue) via setTimeout + AbortController; `settled` flag prevents double-resolve; timer cleared on normal completion so a finished job can never trigger cleanup. On timeout: abort signal -> await cleanup(job) -> outcome "timed-out". enqueue() never rejects — always resolves a typed JobOutcome. Config-decoupled core (plain numbers) + `jobQueueOptionsFromConfig` adapter. Structured start/finish JSON logging via injectable Logger.
- Dedup: since p-queue can't pluck a queued task, a superseded not-yet-started job for the same owner/repo#pr is flagged `cancelled` (resolves "skipped") and the task no-ops when dequeued; map entry deleted only "if still ours". Running jobs are never interrupted (correct per scope). Agent caught + fixed this itself during self-test.
- Exports: JobQueue, JobDescriptor, JobOutcome, JobOutcomeStatus, JobRunner, JobCleanup, Logger, JobQueueOptions, jobQueueOptionsFromConfig.
- Tests (6): concurrency cap (peak <= N), timeout+abort+cleanup -> "timed-out", no-dangling-timer / no post-completion cleanup, start/finish logging, failed-run outcome, dedup/skip.
